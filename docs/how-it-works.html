The big idea

Why a SharedValue pipeline?

In React Native, animating from JS means crossing the bridge every frame — a recipe for jank. This library keeps the whole render loop on the UI thread, where Reanimated worklets run.

❌ The naive way — JS-driven

New data → setState → React re-render → serialize props across the bridge → native redraw. At 60 fps that's 60 round-trips a second fighting the JS event loop. Frames drop the moment JS is busy.

✅ This library — worklet-driven

Data lands in a SharedValue<LiveChartPoint[]>. A useFrameCallback tick reads it, mutates display state, and builds the path into a reused Skia.PathBuilder, detaching a fresh immutable SkPath each frame — all on the UI thread. React barely re-renders at all.

🧵 Worklets

Functions tagged "worklet" run on the UI thread. The tick functions are pure — no hooks, no SharedValues — so they unit-test without Reanimated.

♻️ PathBuilder + detach

A reused Skia.PathBuilder (held in a SharedValue) emits the verbs; detach() yields a fresh immutable SkPath — a new reference Reanimated repaints on, and forward-compatible with Skia 3.0.

📦 Ships TS source

The package main points at src/index.ts; the consumer's Metro + Babel compiles it. dist/ holds only .d.ts types.

The frame engine

What happens inside one tick

Each frame, a pure tick function (tickLiveChartEngineFrame) mutates display state and writes it to SharedValues. Three motions matter most — and you can drive all three below.

1 · The exponential lerp

The raw value jumps when new data arrives. displayValue chases it with a frame-rate-independent ease. This is the library's actual formula: 

const factor = 1 - Math.pow(1 - speed, dt / 16.67);
displayValue += (target - displayValue) * factor;
0.08

● target (raw, jumps)   ● displayValue (smoothed)

2 · Binary-search the visible window

The viewport shows the last timeWindow seconds. Rather than scan every point, the tick binary-searches for the first point ≥ timestamp − displayWindow, then walks forward.

3 · Scan the Y-range, add margin

It scans visible points (plus the live value and reference lines) for min/max, then pads by ~12%. Each bound snaps outward to fit instantly but eases back in — so a spike shows at once yet doesn't leave the axis flickering.

Why pure? The tick functions take a mutable state object + inputs and return void after mutating — no React, no Reanimated. That's what lets liveChartEngineTick.ts be covered by plain Jest tests (90% branches / 95% lines) without a UI-thread round-trip.
The drawing layer

Turning display state into an SkPath

Inside a useDerivedValue, the draw functions map data to screen coordinates and build the path. Two details make it cheap and smooth.

Pixel decimation

A wide timeWindow can pack 1000+ points into a few hundred pixels. Once the line is denser than ~2 points per pixel, buildLinePoints keeps only the lowest and highest sample in each pixel column — so every spike's envelope survives while the vertex count stays bounded by the canvas width. Fewer cubicTo calls, same shape. Toggle it and watch the count.

Fritsch–Carlson monotone spline

A straight polyline looks harsh; a naive cubic overshoots past your data. The monotone spline (drawSpline) guarantees the curve never exceeds the local min/max — smooth, but honest.

The monotone curve hugs the points without inventing peaks that aren't in the data.

The magic sauce

How it holds 60 fps, frame after frame

There's no single trick — it's one architectural decision plus a handful of "do less work per frame" optimizations stacked on top. The decision: the whole render loop lives on the UI thread, and React is kept out of the hot path entirely.

The property that matters when you embed it in a busy app: after mount, the JS thread is essentially idle. The chart keeps animating at 60 fps even while JS is busy — navigation transitions, data fetches, list rendering, your own business logic. The animation never competes with that work, because the two don't run on the same thread.

Architecture — keep JS off the frame path

🔄 One loop, UI thread only

A single useFrameCallback tick reads the data SharedValue, advances display state, and writes it back — all on the UI thread. New data mutates a SharedValue, never setState, so React re-renders roughly once at mount, not 60 times a second. useFrameCallback

🚫 Zero bridge crossings

Everything a frame needs — points, live value, scrub position — already lives in SharedValues on the UI thread. Nothing is serialized across the JS↔native bridge per frame, which is the usual source of React Native chart jank.

🧪 Pure tick = no surprises

The per-frame logic is a pure function (mutable state in, void out) — no hooks, no Reanimated. That keeps it allocation-free and predictable, and unit-tested to 95% lines without a device. liveChartEngineTick.ts

Per frame — do less work, allocate nothing

♻️ Paths built in place

One Skia.PathBuilder per curve is reused every frame; detach() hands back a fresh immutable SkPath, and the flat point buffers ping-pong (A/B). No new arrays or paths per frame means no GC stalls mid-animation. useChartPaths.ts

📉 Draw fewer vertices

buildLinePoints decimates to pixel resolution — at most the lowest + highest sample per pixel column. 1k points or 10k, the vertex count is bounded by canvas width, and every spike's envelope still survives. draw/line.ts

🎯 Batch the overlays

Markers, trade pills, and degen particles render through a single drawAtlas call instead of one draw per glyph — measured to roughly halve CPU on a dense scene. markerAtlas.ts

🌫️ Shade, don't clip

The area-dot fill is a procedural Skia shader (RuntimeEffect), not a per-frame path clip. A clip would force a saveLayer every frame and halve the frame rate to 30 — the shader keeps it free. AreaDotsOverlay

📐 Measure text once

Tooltip and label widths come from a precomputed monospace char-width table, not Skia's measureText — which reshapes glyphs on every call. Scrubbing re-runs that math each frame, so it's a real win. crosshairShared.ts

🧵 Overlays read, never push

Each overlay is a small component subscribed to SharedValues through useDerivedValue. They recompute on the UI thread only when their own inputs change — no parent re-render fanning out across the tree.

Two components

One series, or many

Both compose from the same engine, draw layer, and overlay set — they differ in what enters and what each frame tracks.

LiveChart single-series

data: SharedValue<LiveChartPoint[]> + a live value. Line or candle mode.

  • Momentum-tinted badge & dot (auto-detected from the data)
  • Scrub crosshair with tooltip pill
  • Degen particle burst + screen shake on momentum swings
  • Trade markers & a scrolling buy/sell stream
  • Candlestick mode with live, forming candle
  • Off-axis badges for out-of-range reference lines

LiveChartSeries multi-series

series: SharedValue<SeriesConfig[]> — each with its own id, color, data & live value.

  • Toggleable series via legend chips — opacity eases 0↔1 per series
  • Per-series dots, pulse rings, and inline value labels
  • Y-range scans only the visible series
  • Crosshair returns every series' value at the scrub time
  • Output buffers ping-pong to notify subscribers without allocation
